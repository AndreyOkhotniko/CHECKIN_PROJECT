## Сценарий 1: регистрация и вход (как работает JWT)

```
1. POST /api/auth/register/  { name, email, password, role }
        │
        ▼  сервер: хеширует пароль (bcrypt) → сохраняет User → выдаёт 2 токена
   ответ: { access_token, refresh_token, token_type: "bearer" }

2. Клиент кладёт access_token в заголовок каждого защищённого запроса:
        Authorization: Bearer <access_token>

3. На защищённом эндпоинте Depends(get_current_user):
        - достаёт токен из заголовка
        - verify_token() проверяет подпись (SECRET_KEY) и срок (exp)
        - грузит User из БД по id (sub) → передаёт в эндпоинт

4. Когда access истёк (через 24 часа):
        POST /api/auth/refresh/  { refresh: <refresh_token> }
        → выдаются новые access + refresh.
```

Где это в коде:
- генерация токенов — [security.py](../app/core/security.py),
- проверка токена и загрузка пользователя — [dependencies.py](../app/api/dependencies.py).

**Сроки жизни токенов:** access — 24 часа, refresh — 7 дней (заданы в `security.py`).

## Сценарий 2: постановка штампа (основная фича)

```
Пользователь сканирует QR места в приложении.

1. (опционально) POST /api/stamps/verify/  { qr_code }
        → проверяем, что место существует и активно,
          что штампа ещё нет; отдаём превью (название, картинка печати).
          Сам штамп НЕ сохраняется.

2. POST /api/stamps/  { qr_code }  (или { place_id })
        → находим место,
        → проверяем, что пользователь тут ещё не отмечался,
        → создаём Stamp, увеличиваем user.stamps_count,
        → commit.
   ответ: данные штампа (id, место, дата).
```

Защита от повторного штампа двойная:
1. явная проверка в коде (`select Stamp where user_id & place_id`),
2. **уникальное ограничение в БД** `uq_stamp_user_place` — последний рубеж,
   срабатывает даже при гонке двух одновременных запросов.

## Сценарий 3: офлайн-синхронизация

Если пользователь сканировал QR без интернета, приложение копит чек-ины
локально и потом отправляет их пачкой:

```
POST /api/stamps/offline_sync/  { items: [ { qr_code, scanned_at }, ... ] }
        → по каждому элементу: ищем место, проверяем дубли
          (в т.ч. дубли внутри самого пакета), создаём штамп.
        → каждый штамп пишется в своём savepoint: ошибка одного
          не рушит всю пачку.
   ответ: { successful: [...], errors: [...] }
```

## Сценарий 4: владелец места (роль `obj`)

```
1. POST /api/places/        — создать место (сервер генерит уникальный qr_code).
2. GET  /api/places/{id}/qr/ — получить QR, чтобы распечатать/показать.
3. GET  /api/places/{id}/stats/ — сколько штампов и кто последний отметился.
```
Создание/редактирование/статистика доступны только владельцу (проверка
`place.owner_id == current_user.id`).
