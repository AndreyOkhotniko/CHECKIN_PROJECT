# 2. Архитектура и структура кода

## Слоёная архитектура

Проект следует стандартному для FastAPI разделению на слои. Запрос проходит
сверху вниз:

```
HTTP-запрос
   │
   ▼
[ endpoints ]  app/api/v1/endpoints/*.py
   │   роутеры FastAPI: принимают запрос, проверяют права, вызывают БД
   │   валидируют вход/выход через схемы
   ▼
[ schemas ]    app/schemas/*.py
   │   Pydantic-модели: что приходит в запросе и что отдаём в ответе
   │   (валидация типов, сокрытие полей вроде hashed_password)
   ▼
[ models ]     app/models/user.py
   │   SQLAlchemy ORM: как данные лежат в таблицах БД
   ▼
[ db ]         app/db/session.py
   │   движок и сессии БД (async)
   ▼
PostgreSQL
```

**Почему так:** каждый слой решает одну задачу. Схемы (что отдаём наружу)
отделены от моделей (как храним) — это позволяет, например, никогда не
возвращать хеш пароля клиенту и отдавать публичный список мест без QR-кодов.

## Карта файлов

### Точка входа
- [app/main.py](../app/main.py) — создаёт объект `FastAPI`, подключает все роутеры,
  описывает `lifespan` (события старта/остановки) и эндпоинты `/` и `/health`.

### Конфигурация
- [app/core/config.py](../app/core/config.py) — класс `Settings` (pydantic-settings).
  Читает переменные из `.env` и окружения. Здесь же собираются `DATABASE_URL`
  (async, для приложения) и `SYNC_DATABASE_URL` (sync, для Alembic).
- `.env` — реальные секреты, **в git не коммитится** (см. `.gitignore`).
- [.env.example](../.env.example) — шаблон: скопируй в `.env` и заполни.

### Безопасность / аутентификация
- [app/core/security.py](../app/core/security.py) — хеширование паролей (`bcrypt`
  через passlib) и работа с JWT: `create_access_token`, `create_refresh_token`,
  `verify_token`.
- [app/api/dependencies.py](../app/api/dependencies.py) — зависимость
  `get_current_user`: достаёт JWT из заголовка `Authorization: Bearer ...`,
  проверяет его и возвращает объект `User`. Подключается к защищённым эндпоинтам
  через `Depends(get_current_user)`.

### Доступ к БД
- [app/db/base.py](../app/db/base.py) — `Base = declarative_base()`, общий
  родитель всех ORM-моделей.
- [app/db/session.py](../app/db/session.py) — async-движок, фабрика сессий
  `async_session_maker` и зависимость `get_db()` (отдаёт сессию, откатывает
  транзакцию при ошибке, закрывает в конце).

### Модели (ORM)
- [app/models/user.py](../app/models/user.py) — `User`, `Place`, `Stamp`,
  `Category` (все в одном файле).

### Схемы (Pydantic)
- [app/schemas/user.py](../app/schemas/user.py) — `UserCreate`, `LoginRequest`,
  `UserResponse`, `UserUpdate`, `TokenResponse`, `TokenRefresh`.
- [app/schemas/place.py](../app/schemas/place.py) — схемы мест. Обрати внимание
  на пару `PlacePublicResponse` (без `qr_code`, для публичных эндпоинтов) и
  `PlaceResponse` (с `qr_code`, только владельцу).
- [app/schemas/stamp.py](../app/schemas/stamp.py) — схемы штампов и
  офлайн-синхронизации.

### Эндпоинты (роутеры)
- [auth.py](../app/api/v1/endpoints/auth.py) — `/api/auth`: регистрация, вход, refresh.
- [users.py](../app/api/v1/endpoints/users.py) — `/api/users`: профиль, реферальная ссылка.
- [places.py](../app/api/v1/endpoints/places.py) — `/api/places`: CRUD мест, QR, статистика.
- [stamps.py](../app/api/v1/endpoints/stamps.py) — `/api/stamps`: проверка QR, постановка штампа, коллекция, офлайн-синхронизация.
- [categories.py](../app/api/v1/endpoints/categories.py) — `/api/categories`: список категорий.

### Миграции БД
- [alembic.ini](../alembic.ini) и [alembic/env.py](../alembic/env.py) — настройка
  Alembic. Использует **синхронный** `SYNC_DATABASE_URL` (Alembic не работает с
  async-движком напрямую), поэтому в зависимостях нужен `psycopg2-binary`.

### Инфраструктура
- [docker-compose.yml](../docker-compose.yml) — поднимает PostgreSQL (порт 5432)
  и pgAdmin (порт 5050) для локальной разработки.
- [requirements.txt](../requirements.txt) — Python-зависимости с комментариями.

### Заготовки / пустышки
- `app/core/database.py` — **пустой файл** (артефакт; рабочая логика БД
  находится в `app/db/session.py`). Можно удалить.

## Версии и важные детали

- **SQLAlchemy 2.0 в async-режиме.** Запросы пишутся через `select(...)` +
  `await db.execute(...)`, а не через старый `db.query(...)`.
- **Pydantic v2.** В схемах используется `class Config: from_attributes = True`
  (раньше называлось `orm_mode`), чтобы строить ответ прямо из ORM-объекта
  через `.model_validate(obj)`.
- **Python 3.9** (см. venv). Поэтому, например, `list[PlaceResponse]` работает,
  а вот более новый синтаксис нужно проверять на совместимость.
